# Try — SL API 스키마를 프로젝트마다 안 들고도 그래프에 주입

> 작성일: 2026-06-24
> 대상 작업: `codegraph init`/`init -i` 시 프로젝트 루트 하위에 `.sl`이 감지되면, `fegate_api` 헤더·예제·문서(FFSAPI.md / readmap 등)를 프로젝트에 두지 않아도 **FEGate SL API 스키마를 그래프에 자동 포함/배선**해, 어디서든 API 질의·call graph 해석이 되게 한다.
> 선행: [SL 파서 구현 결과](output/2026-06-24-sl-parser-execute-results.md) — `.sl` 문법 + `.h` HELP_FUN 스텁 + `::` 해석은 **이미 동작**(스텁 노드만 DB에 있으면 해석됨).

## 1. 현황 (As-Is)

| 항목 | 현재 동작 |
|---|---|
| 인덱싱 범위 | `indexAll`이 **단일 `rootDir` 한 곳만** 스캔 (`src/extraction/index.ts:669`, `scanDirectoryAsync(this.rootDir,…)`). 외부/라이브러리 루트 개념 **없음** |
| SL API 해석 | `.sl`의 `dbNode::First()` → ref `dbNode::First` → resolver의 `getNodesByQualifiedName`가 **헤더 스텁 노드**(qualifiedName `dbNode::First`)와 정확 매칭. **단, 그 스텁 노드가 같은 DB에 있어야 함** |
| 스텁 노드 출처 | `fegate_api/*.h`가 **루트 하위에 물리적으로 존재**해야 인덱싱됨 → 없으면 API 콜 전부 미해석 |
| 배포 형태 | codegraph = 공개 범용 `@evespimrose/codegraph` (npm). 문법 wasm은 `dist/`에 동봉(`copy-assets`) |

**핵심 문제**: API 해석 로직은 위치 무관(스텁 노드 존재 여부에만 의존)인데, **스텁의 출처인 헤더를 매 SL 프로젝트가 루트 안에 들고 다녀야** 한다. readmap/FFSAPI.md/priority 파일은 [선행 Try 결론](../raonx-ai-workflow/docs/Try_SLGrammarParser_Readmap.md)대로 **파서 도입 후 실효성 0** — 스텁이 대체. 즉 진짜 필요한 건 `fegate_api/*.h`(스키마 원천)뿐이고, 그조차 매 프로젝트 동봉은 낭비/표류.

## 2. 제안 구조 (To-Be) — 비용 사다리

| 옵션 | 스키마 위치 | 주입 방식 | 핵심 결함/특성 |
|---|---|---|---|
| **A. 현상유지** | 매 프로젝트 루트 `fegate_api/` | 일반 인덱싱 | 매 프로젝트 헤더 동봉(들고 다님) |
| **B. 패키지 동봉 + init 시 복사·인덱싱** | **codegraph 패키지 내부** | SL 감지 시 헤더를 `.codegraph/`로 복사 후 인덱싱 | **공개 패키지에 독점 벤더 데이터 적재**·버전 동결 |
| **C. 사전계산 그래프 프래그먼트 주입** | codegraph 패키지 내부(직렬화 노드) | SL 감지 시 ~1,713 스텁 노드를 DB에 머지 | 동상 + **신규 머지 서브시스템**(노드ID 안정성·dedup·external 표시·watch 상호작용) |
| **D. 설정형 외부/라이브러리 루트(범용)** ⭐ | **머신당 1곳 공유** `…/fegate_api` (env/config 지정) | init이 그 루트도 스캔 → `external` 표식으로 머지 | 범용·탈독점·버전 추적. 인덱싱 재사용 |

```
탈독점·범용성   A < B ≈ C < D        (D만 codegraph를 벤더-중립으로 유지)
"안 들고 다님"  A < B < C ≈ D        (B/C/D 모두 per-project 동봉 제거)
구현·유지비용   A < D < B < C        (C가 최고난도; D는 인덱싱 재사용)
버전 표류위험   A·D 낮음  /  B·C 높음 (동봉 스키마는 릴리스 시점에 동결)
```

**D 메커니즘(최소)**: 외부 루트의 `*.h`를 기존 `SlHeaderExtractor`로 인덱싱 → 동일 DB에 `dbNode::First` 스텁 노드 생성 → **해석은 이미 동작**(실측: 스텁이 있으면 `::` 2,544건 해석). 노드에 `external`/library provenance만 부여해 "내 코드" 지표·watch에서 분리. **빠진 조각은 '루트 밖 헤더를 그래프에 넣는 경로' 하나뿐.**

## 3. 장단점 비교

### 3-A. 아키텍처 명확성
- **B·C 치명**: 공개 범용 도구(`@evespimrose/codegraph`)에 **한 벤더의 독점 API 스키마**를 적재 → 도메인 중립성 위반, 패키지 비대, "범용 도구가 특정 SDK에 의존"하는 **역방향 결합**. codegraph의 언어-범용 철학과 충돌.
- **D 우월**: "추가 라이브러리 루트를 external로 인덱싱"은 **언어 불문 범용 기능**(stdlib·임의 SDK 헤더에도 적용). FEGate 스키마는 도구 밖(설정)에 머묾 → 결합 방향 정상(도구→범용, 벤더데이터는 사용자 소유).
- **자동 트리거(SL 감지→스키마 당김)**: 편의지만 "`.sl`=FEGate" 가정의 마법적 결합. **명시 설정(opt-in)이 더 깔끔**(머신당 1회 지정으로 "안 들고 다님" 목표 동일 달성).

### 3-B. 컴파일·빌드/인덱싱 영향
- **B**: 헤더 38개(소용량) 동봉이지만 **벤더 데이터 적재 선례**를 염. init마다 복사+인덱싱 단계 추가.
- **C**: 직렬화 그래프 + **신규 머지 코드**(노드ID 충돌/중복/외부표식). 최고 난도.
- **D**: `scanDirectoryAsync`를 멀티루트로 확장(또는 2차 인덱싱 패스) + external 태깅. **실측상 1,678 스텁 인덱싱 <1s**(저비용 재인덱싱). 기존 파이프라인 재사용.

### 3-C. 현재 작업과의 연관성
- 방금 출시한 SL 파서(`SlHeaderExtractor`+`slExtractor`+`::` 해석)의 **자연스러운 다음 레이어** — "헤더 없이도 그 해석을 쓰게" 하는 것. 강한 연속성.
- 단 이건 **버그픽스가 아니라 별도 기능(external-roots)**. SL 파서 자체는 완결(스텁만 있으면 해석됨). 타이밍: SL 파서 안정화 후 착수가 합리적.

### 3-D. 단기 비용 (마이그레이션 리스크)
- **B**: 코드 적지만 **독점-인-패키지**가 하드 블로커(배포 정책·라이선스).
- **C**: 높음(직렬화 포맷 + 머지 + 표류 처리 + sync/watch 상호작용 회귀 위험).
- **D**: 중간(멀티루트 스캔 + external 노드 종류 + 설정 로더). 가장 무후회. 인덱싱 재사용.

### 3-E. 장기 유지보수
- **B·C**: 동봉 스키마는 **릴리스 시점에 동결** → 사용자의 실제 FEGate 엔진 버전과 **표류**(조용한 오해석). API 변경마다 재번들 + codegraph 릴리스를 벤더 주기에 결박.
- **D**: 스키마가 사용자의 공유 헤더(=그들의 엔진 버전)를 추적 → 탈동결. **범용 external-roots는 모든 SDK에 분할상환**(가장 건강).
- 자동 트리거: 확장자-스니핑→벤더 가정의 취약성 영구. 명시 설정이 유지보수 우위.

## 4. 순환 의존성·방향 위험 분석

```
[B·C 위험] 공개 범용 도구 ──의존→ 독점 FEGate 스키마(패키지 내부)
            → "일반이 특정에 의존"하는 역방향. 배포·라이선스·표류 3중 부채.
            ⊕ 동봉 그래프(C)는 동결 아티팩트 ↔ 라이브 그래프 간 노드ID/dedup 결합
              = 새 표류원(선행 Try가 경고한 미러·readmap 표류와 동형).

[D 안전] codegraph(범용) ──스캔→ 사용자 지정 외부 루트(사용자 소유 데이터)
            → 방향 정상(도구는 중립, 벤더데이터는 설정으로 외주). 파생물 0.
            FEGate 헤더 = 머신당 1 SSOT. 표류원 없음.
```

원칙: **독점 벤더 데이터를 범용 배포물에 굽지 말 것.** "안 들고 다님"은 *패키지 동봉*이 아니라 *머신당 1곳 공유 + 설정 참조*로 달성해야 정합.

## 5. 종합 평가 및 추천

### 5-1. 정합성 판정
- **목표(헤더 없이 API 해석)는 가치 높음** + 방금 만든 SL 파서의 정당한 다음 레이어. 해석 로직은 위치 무관이라 **"스텁 노드를 DB에 넣는 경로" 하나만 추가하면 됨** — 기술 난도 낮음(D 기준).
- **글자 그대로의 "패키지에 동봉해서 자동 주입"(B/C)은 부정합**: 공개 범용 codegraph에 독점 FEGate 스키마 적재 = 도메인 결합·버전 표류·배포 부채. **반대.**
- ∴ 정합적 최적 = **범용 external/library-roots 기능**(D). FEGate 헤더는 도구 밖(머신당 1 공유 + env/config)에서 참조.

### 5-2. 추천 형태 (진행 시 단계)
1. **무후회 즉답(코드 0)**: `fegate_api/`를 머신당 1곳에 두고, 각 SL 프로젝트는 그 경로를 **루트 하위 심볼릭/정션**으로 연결하거나 `.codegraphignore` 화이트리스트로 포함. → 오늘 당장 "복사본 난립" 제거. (여전히 per-project 링크 1개는 남음)
2. **본 기능(D)**: `codegraph init --external <path>` 또는 `.codegraph/config.json {"externalRoots":[...]}` 또는 `CODEGRAPH_EXTERNAL_ROOTS` env. 인덱싱이 외부 루트도 스캔 → 노드에 `external` provenance. **per-project 설정 0**(머신/유저 레벨 1회). → "안 들고 다님" 완성.
3. **선택 편의**: SL 감지 시 미설정이면 "외부 SL API 루트 지정?" **안내만**(자동 당김 금지 — opt-in 유지).
4. **배포 변형(원할 때)**: FEGate 스키마를 **별도 동반 패키지**(`@evespimrose/codegraph-fegate` 데이터팩)로 설치형 제공 — 코어는 범용 유지, 스키마는 옵션. (B의 "설치형" 정합 버전)

### 5-3. 한 줄 결론
> "헤더 없이 API 해석"은 **만들 가치가 있고 기술적으로 쉽다(스텁 노드만 DB에 있으면 됨)** — 단 그 수단은 **공개 패키지 동봉(B/C)이 아니라 범용 external-roots 설정(D)**. 그래야 codegraph는 벤더-중립으로 남고, FEGate 헤더는 머신당 1 SSOT로 표류 없이 모든 SL 프로젝트에 적용된다. 자동 트리거는 안내까지만, 주입은 명시 opt-in.
