# Try — init/index 제외 규칙을 .gitignore 동기화에서 별도 ignore 파일로 전환

> 작성일: 2026-06-15
> 대상 작업: codegraph가 `init`/`index` 시 파일/폴더/확장자를 제외하는 로직을, 현재의 "프로젝트 루트 .gitignore 동기화" 방식에서 **별도 ignore 파일**(예: `.codegraphignore`) 기반으로 변경

---

## 1. 현황 (As-Is)

### 1.1 파일 구조

제외 로직은 전부 `src/extraction/index.ts` 한 곳으로 수렴하며, 세 계층으로 작동한다.

| 계층 | 위치 | 적용 범위 |
|------|------|----------|
| ① 내장 기본 제외 | `DEFAULT_IGNORE_DIRS` / `DEFAULT_IGNORE_PATTERNS` (`src/extraction/index.ts:145–189`) | git 여부·tracked 여부 무관, 항상 적용 (issue #407) |
| ② 루트 `.gitignore` 병합 | `buildDefaultIgnore(rootDir)` (`src/extraction/index.ts:198–207`) | 루트 `.gitignore`를 ①에 머지 → `!vendor/` 같은 negation으로 기본값 override 허용 |
| ③ 중첩 `.gitignore` (디렉토리별) | git 경로: `git ls-files`가 네이티브로 처리 · 비-git 경로: `scanDirectoryWalk`의 `loadIgnore` (`:434–444`, 디렉토리별 scoped matcher 적층) | 모든 하위 디렉토리 레벨 |

스캔 진입점 `scanDirectory` (`:362`)는 두 경로로 갈린다.
- **git 빠른 경로** `getGitVisibleFiles` (`:265–300`): `git ls-files`(tracked+untracked-not-ignored)로 목록을 얻은 뒤 **그 결과를 다시 `buildDefaultIgnore`로 필터링**(`:296–297`). 임베디드 git repo 재귀·서브모듈 처리 포함.
- **비-git 워크 경로** `scanDirectoryWalk` (`:417–531`): 파일시스템 직접 순회. base matcher를 `buildDefaultIgnore`로 시드(`:529`)하고 중첩 `.gitignore`를 디렉토리별로 적층.

`buildDefaultIgnore`(export됨)의 **소비처는 3곳**:
1. `getGitVisibleFiles` (`src/extraction/index.ts:296`)
2. `scanDirectoryWalk` (`src/extraction/index.ts:529`)
3. `FileWatcher` (`src/sync/watcher.ts:173`) — watch 범위와 index 범위를 일치시키기 위함

별도로 **문서 스캐너** `src/docs/scan-files.ts`는 자체 `SKIP_DIRS`(8개) + `git ls-files --exclude-standard`를 쓰며 `buildDefaultIgnore`를 **사용하지 않는다**(잠재적 4번째 변경 지점).

### 1.2 병목·문제 정량화 (수치 포함)

- 내장 기본 제외 디렉토리 ≈ **62개** + glob 패턴 **3개**(`*.egg-info/`, `cmake-build-*/`, `bazel-*/`) = 약 65개 패턴.
- `buildDefaultIgnore` 소비처 **3곳**, `scanDirectory`/`scanDirectoryAsync` 호출 site **4곳**(`:615`, `:649`, `:1384`, `:1532`).
- `scanDirectory`를 직접 검증하는 테스트 assertion **약 13건**(`__tests__/extraction.test.ts` 10건, `__tests__/security.test.ts` 3건).
- 현재 `.codegraphignore`를 읽는 코드는 **0건**(grep 확인: 소스에 미존재, CHANGELOG·분석문서에만 언급).

### 1.3 결정적 발견 (존재 시)

**(A) 사용자의 전제는 부분적으로만 맞다.** 현재는 단순한 "루트 .gitignore 동기화"가 아니다. 실제로는 ① 내장 기본 65패턴 + ② 루트 .gitignore 병합 + ③ **git이 네이티브로 수행하는 모든 레벨 .gitignore 해석**의 합성이다. "동기화"의 본체는 codegraph 코드가 아니라 **`git ls-files`의 내장 동작**이다.

**(B) 단일 chokepoint가 이미 존재한다 — 빼는(subtract) 방향은 거의 공짜.** git 빠른 경로조차 `git ls-files` 결과를 `buildDefaultIgnore`로 **재필터링**(`:296–297`)한다. 따라서 `.codegraphignore`를 **`buildDefaultIgnore` 안에서 읽도록 1개 함수만 고치면** git 경로·워크 경로·watcher 3곳 모두 자동으로 "추가 제외"를 얻는다. 이것이 이 작업의 가장 큰 레버리지다.

**(C) 그러나 "되살리기(re-include)"는 git 경로에서 구조적으로 불가능하다.** `.codegraphignore`의 negation(`!build/keep.js`)으로 .gitignore된 파일을 그래프에 **다시 포함**하려는 시나리오는, `git ls-files`가 그 파일을 애초에 출력하지 않으므로 `buildDefaultIgnore` 필터로는 절대 살릴 수 없다(필터는 빼기만 함). 이 경우 git 빠른 경로를 **포기하고** 풀 파일시스템 워크로 전환해야 하며 → 속도·서브모듈·임베디드 repo 정확성 손실. 이것이 "별도 파일로 완전 대체"의 핵심 비용이다.

## 2. 제안 구조 (To-Be)

### 2.1 구조 다이어그램 (현재 vs 제안)

```
[현재]
scanDirectory ─┬─ git 경로: git ls-files ─→ buildDefaultIgnore(기본+루트.gitignore) 필터
               └─ 워크 경로: walk + buildDefaultIgnore 시드 + 중첩 .gitignore 적층
                                   ↑
                         FileWatcher 도 동일 matcher 공유

[제안 A — 가산(추가 제외)·하위호환]  ← 권장
buildDefaultIgnore(rootDir):
   ignore(기본 65패턴)
     .add(루트 .gitignore)          ← 유지(또는 플래그로 끔)
     .add(루트 .codegraphignore)    ← 신규 1줄. 3 소비처 전부 자동 반영
   → git·워크·watcher 모두 "추가 제외" 즉시 동작 (빼기 전용)

[제안 B — 완전 대체]  ← 고위험
respectGitignore=false 시:
   git 빠른 경로 사용 안 함 → 풀 워크 + 기본패턴 + .codegraphignore(중첩 포함)만 적용
   → re-include 가능해지나 속도/서브모듈/임베디드 repo 정확성 손실
```

### 2.2 새 파일 목록

- (소스 신규 없음 — 권장안은 기존 `buildDefaultIgnore` 1함수 확장)
- 런타임 산출물: 사용자 프로젝트 루트의 **`.codegraphignore`** (gitignore 문법). 위치·공유 여부는 §6 미결.
- (선택) `codegraph init` 시 주석 달린 `.codegraphignore` 스캐폴드 생성 — `src/directory.ts`의 `.codegraph/.gitignore` 자동 생성 패턴(`:83–105`) 재사용.

### 2.3 핵심 API·알고리즘 개념 코드 (의사 코드 수준)

```ts
// src/extraction/index.ts — buildDefaultIgnore 확장 (제안 A)
export function buildDefaultIgnore(rootDir: string, opts?: { respectGitignore?: boolean }): Ignore {
  const ig = ignore().add(DEFAULT_IGNORE_PATTERNS);          // ① 내장 기본 (항상 유지 권장)
  if (opts?.respectGitignore !== false) {                    // ② 루트 .gitignore (플래그로 끌 수 있게)
    tryAdd(ig, path.join(rootDir, '.gitignore'));
  }
  tryAdd(ig, path.join(rootDir, '.codegraphignore'));        // ③ 신규: 마지막에 add → 최우선 override
  return ig;
}
// 순서 = ignore 매처의 last-match-wins. .codegraphignore를 맨 뒤에 두면
// 기본값·gitignore 둘 다 negation으로 덮을 수 있음(단 git 경로 re-include 한계는 §1.3-C).
```

## 3. 장단점 비교

### 3-A. 아키텍처 명확성
- (+) 제외 정책 단일 chokepoint(`buildDefaultIgnore`)가 이미 존재 → 가산안은 3 소비처에 일관 반영, 분기 없음.
- (+) "그래프 범위"와 "git 추적 범위"를 의도적으로 분리 → 커밋된 생성물 제외, 또는 비-git 폴더 정책 표현이 명시적.
- (−) `.gitignore`/`.codegraphignore`/내장기본 3중 소스의 **우선순위**를 문서화하지 않으면 혼란. 중첩 `.gitignore`까지 합치면 4중.

### 3-B. 컴파일·빌드 영향
- 가산안: 타입 변경 없음(옵션 인자 추가는 후방호환). `copy-assets`·wasm·schema 영향 없음. 빌드 영향 ≈ 0.
- 완전 대체안: git 경로 분기 신설 → `getGitVisibleFiles` 시그니처/호출부 변경, 빌드는 무난하나 런타임 경로 재검증 필요.

### 3-C. 현재 작업과의 연관성
- 현재 브랜치 `Obsidian-Graph-Vector-DB`(문서·벡터DB 연계)와 직접 충돌 없음. 단 `src/docs/scan-files.ts`가 별도 ignore를 갖고 있어, "문서 인덱싱에도 `.codegraphignore` 적용?" 여부를 이 작업 범위에 넣을지 결정 필요(§6).

### 3-D. 단기 비용 — 고위험 / 중위험 / 저위험

| 위험도 | 항목 | 근거 |
|--------|------|------|
| **고위험** | git 빠른 경로 포기(완전 대체안) | `git ls-files` 기반 모든-레벨 .gitignore·서브모듈·임베디드 repo 재귀(`:214~`)를 잃음. issue #407이 세운 "git-first" 전략 정면 회귀. |
| **고위험** | re-include 기대 불일치 | `.codegraphignore` negation으로 gitignore된 파일 부활은 git 경로에서 불가(§1.3-C). 미문서화 시 "내 패턴이 안 먹는다" 버그 리포트 직결. |
| **중위험** | 내장 기본 65패턴을 "대체"로 오해 | "별도 파일로 변경"을 기본 제외 제거로 해석하면 node_modules 등 재유입 → 그래프 폭발·성능 급락. 기본값은 **항상 유지** 명시 필요. |
| **중위험** | watcher/index/docs 스캐너 동기화 | 3(+1 docs) 소비처가 동일 정책을 봐야 함. 가산안은 chokepoint 덕에 자동 충족하나, docs 스캐너는 별도 패치 필요. |
| **중위험** | 테스트 갱신 | `scanDirectory` assertion ~13건 + 신규 `.codegraphignore` 케이스. CLAUDE.md house-rule상 동작 변경엔 커버리지·CHANGELOG 필수. |
| **저위험** | 크로스플랫폼 경로 | `normalizePath`·`ignore`(posix) 변환 기존 플러밍 재사용 → 추가 위험 낮음. Windows 게이트 테스트만 확인. |

### 3-E. 장기 유지보수
- 가산안은 표면적이 작아 유지보수 부담 낮음. 다만 "ignore 소스가 몇 개냐"는 사용자 인지 비용이 늘어남 → 단일 진실 문서(우선순위표) 1개로 상쇄 권장.
- 완전 대체안은 git 경로/워크 경로 2벌의 .gitignore 해석을 영구히 직접 떠안음(git이 공짜로 해주던 일) → 장기 부채 큼.

## 4. 순환 의존성·방향 위험 분석

- 의존 방향: `sync/watcher.ts → extraction`(이미 `buildDefaultIgnore` import). `.codegraphignore` 읽기를 `buildDefaultIgnore` **내부**에 두면 방향 불변, 신규 순환 **없음**.
- 별도 `src/config/` 모듈을 신설해 extraction이 import하는 설계는, config가 extraction을 역참조하지 않는 한 무순환(현재 그럴 이유 없음). 단 굳이 신설 시 chokepoint가 둘로 갈려 3-A 이점이 희석됨 → 비권장.
- `src/docs/scan-files.ts`는 extraction에 의존하지 않는 독립 스캐너 → 여기에 `.codegraphignore`를 적용하려면 공용 로더를 extraction에서 export해 docs가 import(방향 안전).

## 5. 종합 평가 및 추천

### 결론 (한 문단)
"별도 ignore 파일"을 **기존 .gitignore에 얹는 가산(추가 제외) 레이어**로 정의하면, `buildDefaultIgnore` 단일 함수에 `.codegraphignore` 로드 1줄을 더하는 것만으로 git·워크·watcher 3경로 전부에 일관 반영되는 **저위험·고효율** 변경이다(§1.3-B). 반대로 사용자의 의도가 "gitignore를 완전히 대체"하거나 "gitignore된 파일을 그래프에 되살리기"라면, git 빠른 경로(`git ls-files`)를 포기해야 하므로 issue #407의 git-first 전략을 회귀시키는 **고위험** 작업이 된다(§1.3-C, §3-D). 따라서 착수 전 **"추가 제외만 필요한가, 완전 대체/되살리기까지 필요한가"**를 사용자에게 먼저 확정해야 한다 — 이 한 가지 답이 난이도를 1줄 vs 경로 재설계로 가른다.

### 단계별 추천 순서
- **Phase 0 (의사결정·선행 필수)**: 사용자에게 의도 확정 — ⓐ 추가 제외만(가산) / ⓑ gitignore 무시 토글까지 / ⓒ 되살리기(re-include)까지. + 우선순위(내장기본 vs .gitignore vs .codegraphignore) + 파일 위치(루트 공유 vs `.codegraph/` 로컬) + docs 스캐너 적용 여부.
- **Phase 1 (가산안, ⓐ면 종료)**: `buildDefaultIgnore`에 `.codegraphignore` 로드 추가(맨 뒤 add). 내장 기본은 유지. `extraction.test.ts`/`security.test.ts`에 제외·negation 케이스 추가, CHANGELOG `[Unreleased]` 기재.
- **Phase 2 (ⓑ 필요 시)**: `respectGitignore` 옵션을 `buildDefaultIgnore`/`scanDirectory`/CLI 플래그로 노출. 끄면 루트 .gitignore 병합만 생략(중첩·git ls-files 동작과의 상호작용 문서화).
- **Phase 3 (ⓒ 필요 시, 고위험)**: re-include 모드 — git 경로를 풀 워크로 강제 전환하는 별도 경로 신설 + 성능/정확성 회귀 측정(README 7-repo 대조군). 충분한 수요 확인 전 보류 권장.
