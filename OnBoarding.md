# 🚀 Trae-Claude Context & Dictionary-First 온보딩 가이드

> **"단 15토큰으로 수천 행의 코드베이스를 탐색하고 제어하는 초효율적 AI 협업 아키텍처"**
> 본 가이드는 요구사항 가상화 파이프라인(`trae-claude-context-workflow`)과 사전 기반 코드 경로 탐색(`Dictionary-First Grep Workflow`)을 유기적으로 통합하여, AI 에이전트의 컨텍스트 왜곡과 토큰 폭탄을 원천 차단하기 위한 프로젝트 공식 지침입니다.

---

## 1. 아키텍처 배경 및 핵심 문제 의식 (Why)

### ❌ 기존 방식의 한계 (Raw Paste & Bash Grep)
1. **컨텍스트 선형 누적:** 유저가 날것의 피드백을 직접 붙여넣으면 대화가 거듭될수록 이전 이력이 통째로 다시 전송되어 토큰 소모량이 기하급수적으로 폭증합니다.
2. **Bash 탐색 복리 소모:** Claude가 특정 코드 위치를 찾기 위해 `find`, `grep`, `ls` 등의 터미널 명령을 수행하면 수백 행의 텍스트 오염이 발생하며, 장기 세션 진행 시 동일한 탐색 노이즈가 매 턴 반복 전송되는 최악의 토큰 낭비가 발생합니다.

### 🎯 통합 워크플로우의 목표
- **요청 구조화(Structuring)를 외부 AI로 격리하여 Claude의 입력 페이로드를 압축합니다.**
- **터미널을 통한 소스코드 Grep/Find 행위를 완전히 금지하고, 고도로 정제된 `manage/dictionary.md` 단 한 장으로 모든 코드의 위치(Pathfinding)를 식별합니다.**

## 2. 통합 아키텍처 데이터 흐름 (Data Flow)

이 시스템은 **[외곽 격리 레이어(Trae)] ➔ [영구 정제 레이어(Local DB/MD)] ➔ [핵심 추론 레이어(Claude Code)]**의 3단계 가상화 파이프라인으로 동작합니다.
```text
[개발자의 날것의 피드백 / 이슈 로그]
│
▼
┌────────────────────────────────────────────────────────┐
│ 1단계: 외곽 격리 레이어 (Trae Context-Sharer Skill)      │
│  - 비정형 요구사항을 고정 밀도의 기능 블록(BLK) 스펙으로 구조화   │
│  - docs/contextmd/cxt[N].md 파일 생성 및 규격 검증        │
│  - 클립보드에 명령어 자동 복사: /doc-context [경로]         │
└───────────────────────┬────────────────────────────────┘
│ (클립보드 데이터 Ctrl+V)
▼
┌────────────────────────────────────────────────────────┐
│ 2단계: 영구 정제 레이어 (manage/dictionary.md)          │
│  - 항상 메모리 최상단에 로드되어 있는 프로젝트 마스터 맵     │
│  - "어떤 파일이 어디에, 어떤 BLK 의존성을 갖고 있는가?" 정보 공급 │
│  - 터미널 Grep 명령어를 단 30토큰 미만의 사전 조회로 대체    │
└───────────────────────┬────────────────────────────────┘
│ (Grep/Find 명령어 완전 격리 및 대체)
▼
┌────────────────────────────────────────────────────────┐
│ 3단계: 핵심 추론 레이어 (Claude Code & Hooks)           │
│  - Read 훅 작동: cxt[N].md 로드 시 관련 BLK 맥락 주입      │
│  - Write 훅 작동: 소스 수정 시 dictionary.md 실시간 동기화   │
│  - 초소형 역사적 발자국(History Footprint)만으로 코드 작성    │
└────────────────────────────────────────────────────────┘
```

## 3. 정량적 토큰 감량 및 효율성 비교 (Token Math)

본 통합 워크플로우를 전면 채택할 경우 발생하는 구체적인 비용 및 성능 개선 지표입니다.

### 📊 컴포넌트별 소모 토큰 비교

| 비교 항목 | 기존 생짜 작업 (Raw Paste + Bash) | 통합 아키텍처 워크플로우 | 토큰 절감률 |
| :--- | :--- | :--- | :--- |
| **단일 요구사항 파싱** | 180 ~ 300 토큰 | **~15 토큰** (명령어만 주입) | **91.6% 절감** |
| **단일 파일/클래스 탐색** | 600 ~ 1,200 토큰 (Grep 출력물) | **~30 토큰** (사전 핀포인트) | **95.0% ~ 97.5% 절감** |
| **15턴 세션 누적 비용** | 20,000 ~ 30,000 토큰 (복리 폭발) | **2,000 ~ 4,000 토큰** | **~85.0% 절감** |

* **손익분기점(Break-even):** 단 **1.45턴**. 두 번째 대화를 주고받는 순간부터 통합 워크플로우가 무조건 비용 및 컨텍스트 효율 면에서 압도적으로 승리합니다.

## 5. 아키텍처의 장단점 및 유지보수 분석 (Trade-offs)

### 👍 장점 (Benefits)
1. **컨텍스트 오염 제로(Context Purity):** 무거운 터미널 로그와 인간의 난잡한 산문이 대화 창에 들어오지 않으므로 Claude의 어텐션 분산이 일어나지 않으며, 환각(Hallucination)율이 0%에 수렴합니다.
2. **극단적 비용 절감:** 컨텍스트가 압축되어 고정밀 상태를 유지하므로 API 비용이 기존 대비 **80% 이상 영구 절감**됩니다.
3. **요구사항 추적성(Traceability):** 모든 기획 변경 및 피드백이 파일 형태로 Git에 기록되므로, 언제든 특정 시점의 개발 컨텍스트를 100% 복구하여 재작업할 수 있습니다.

### 👎 단점 및 한계 (Limitations)
1. **추가적인 런타임 오버헤드:** 인간 개발자가 Claude에게 직접 말을 걸기 전, 외곽 AI(Trae 등)를 거쳐서 스펙 파일을 생성하고 복사해야 하는 단계적 비용(Friction)이 1단계 추가됩니다.
2. **동기화 오염 위험:** 소스코드가 대규모로 변경되었음에도 `manage/dictionary.md` 내부의 인덱스(BLK 태그)를 제때 갱신하지 않으면, Claude가 구버전 소스 경로를 참조하여 길을 잃을 수 있습니다.

### 🛠️ 유지보수 방침 및 가비지 컬렉션 (GC)
- **자동화 훅(Hooks) 적극 활용:** 사람이 직접 사전을 고치는 실수를 방지하기 위해 코드 변경(`Write`/`Edit`)이나 커밋(`post-commit`) 시 훅 스크립트가 실행되어 사전 동기화 여부를 검증 메시지로 띄웁니다.
- **계층형 분할 기준:** 프로젝트 규모가 커져 단일 `dictionary.md` 파일이 **500행을 초과**하는 시점이 오면, 서브시스템별(예: `dict-pathfinding.md`, `dict-gameloop.md`)로 사전을 레이어 계층형 분할 처리합니다.

## 6. 서브시스템 확장 요소 (Future Extensibility)

현재 이 시스템은 알파 단계(`alpha-0.001-trae`)로 핵심 모듈 위주로 적용되어 있으며, 향후 다음과 같은 방향으로 시스템을 확장해 나갑니다.

1. **서브시스템 인덱스 전면 확장 (Phase 6-A):** 현재 적용된 모듈 외에 게임 루프(Gameloop), 맵 시스템(Map), 유닛 제어(Units) 등 전체 핵심 서브시스템으로 BLK 태그 인덱스 구조를 수평 확장합니다.
2. **가상화 검증 훅 고도화 (Phase 5-A):** `validate-cxt.ps1` 스크립트를 Trae 훅 레이어에 완전히 정착시켜, 생성된 `cxt[N].md` 파일에 유효한 `[BLK-XXX]` 식별 태그가 누락되었을 경우 클립보드 복사 단계를 원천 차단(Exit 1)하는 강제 무결성 필터를 가동합니다.
3. **Claude Code 전용 `/cs` Skill 개발 (Phase 6-C):** 외부 AI 독립성을 확보하기 위해, Trae의 도움 없이 Claude Code 터미널 내부에서 자체적으로 비정형 텍스트를 파싱하여 스펙 파일을 자동 생성하는 래핑 도구를 빌드할 예정입니다.

## 7. 신규 팀원 온보딩 행동 지침 (Quick-Start)

### 🏁 1단계: 개발 세션 시작 및 환경 점검
1. 터미널을 열고 Claude Code를 구동하면 글로벌 플러그인과 로컬 `.claude/hooks/dict-blk-announce.sh`가 연쇄 트리거되어 프로젝트 내부의 사전 상태와 활성 BLK 목록을 컨텍스트에 자동 로드합니다.
2. Claude 대화창에 아래 명령어를 입력하여 환경이 연동되었는지 확인합니다.
   > **인간:** `/doc-context` 스킬이랑 `dictionary.md` 인덱스가 정상적으로 인식되고 있어?

### 🛠️ 2단계: 신규 피드백/기획 변경사항 처리 프로세스
50토큰 이상의 복잡한 요구사항이나 기획 변경 요청이 들어오면 **절대 Claude 창에 생짜 텍스트로 붙여넣지 마십시오.**
1. **Trae**를 켜고 요구사항을 던지며 지시합니다: 
   ```text
   이 내용을 기반으로 cxt130.md를 구조화하고 Claude 핸드오버를 준비해 줘.
   [날것의 기획서/피드백 내용 복사]
   ```
2. Trae가 사양을 정제하고 클립보드에 명령어를 넣어주면, Claude 창으로 돌아와 Ctrl+V (/doc-context docs\contextmd\cxt130.md) 후 엔터를 누릅니다.

3. Claude가 사전 인덱스를 기반으로 단 30토큰 이내의 비용으로 핀포인트 코딩을 완수하는 것을 확인합니다.
