## 1. 현황 (As-Is)

### 1.1 파일 구조 — 세 개의 분리된 검색 평면

| 평면 | 저장소 | 인덱싱 대상 | 진입점 | CLI 노출 |
|---|---|---|---|---|
| **렉시컬(이름)** | `nodes_fts` (FTS5) | `name, qualified_name, docstring, signature` ([schema.sql:97-121](../../src/db/schema.sql)) | `searchNodes()` | ✅ `query`, `context` seed |
| **시멘틱(본문)** | `mdast_vectors`(vec0) + `mdast_metadata.content_summary` | 제목+요약 임베딩(MiniLM 384d) ([search.ts:111](../../src/docs/search.ts)) | `searchDocs()` | ❌ **MCP `codegraph_docs` 전용** |
| **그래프(링크)** | `mdast_metadata.doc_links`(JSON) / `edges`(materialized) | 위키링크·백링크 ([search.ts:262 `findBacklinks`](../../src/docs/search.ts)) | `findBacklinks()` / `callers`·`callees` | ❌ backlinks는 **MCP `codegraph_backlinks` 전용**; `callers`만 CLI |

### 1.2 병목·문제 정량화

**결정적 사실 A — query는 본문을 못 본다.** CLI `query <search>` → `cg.searchNodes()` ([codegraph.ts:910](../../src/bin/codegraph.ts)) → `nodes_fts`. `doc` 노드는 **파일명만** FTS에 들어가고 본문은 `nodes_fts`에 없음 → 본문-시멘틱 쿼리는 구조적으로 `[]`.

**task1.md(WIKI: 146 doc 노드, 666 엣지) 관측:**
- 빈 `query` 호출 **≈10+회** (3개 배치 6+4+4가 전부 `[]`). 예: `"맥락 오염"`→`[]` (노드 `맥락_오염_방지` 존재함에도), 단일어 `"토큰"`·`"compression"`까지 `[]`
- 그 후 `callers "Token_허브"` → 백링크 4개 정상 → 선회
- 에이전트 자가결론(task1.md:108): *"query는 이름만, callees/callers/impact가 실제 그래프 순회"*
- 낭비: 빈 쿼리 10여 회 × (왕복+재추론). 토큰·벽시계 손실 (로그 토큰 미기록 → **추정**)

**결정적 사실 B — BLADE는 그래프 평면이 비어있다.** `D:\Unity\BLADE\.codegraph\codegraph.db` 실측:
- `nodes: 0`, `edges: 0`, `mdast_metadata: 42` (그 중 23개 `doc_links` 보유)
- 즉 링크 데이터는 `mdast_metadata.doc_links`(JSON)에만 있고 `edges` 테이블엔 0개
- → `findBacklinks`(mdast 직접 스캔)는 동작하지만, `callers`/`callees`(edges 순회)는 **BLADE에서 전부 무효**
- BLADE 로드맵 §11.2 동기화 검증 `SELECT COUNT(*) FROM edges >= 1` → **실패** (= §12.3 "영구기관이 죽었다" 조건)

**결정적 사실 C — 한글 파일명 인덱싱 누락.** `query "맥락 오염"`이 `맥락_오염_방지`를 못 찾는 1차 원인은 FTS5 unicode61 토크나이저의 한글+언더스코어 처리. BLADE 로드맵이 이미 "비-ASCII 누락"을 실측 결함으로 명시(§12.1.1, 부록 B `docs/codegraph-analysis.md`)하고 **ASCII 슬러그 마이그레이션**을 1순위 권고.

### 1.3 결정적 발견

1. **task1.md 에이전트는 Bash CLI로 구동** → 시멘틱·백링크·MCP 스티어링 채널에 닿지도 못함. 이미 구현된 `searchDocs`/`findBacklinks`가 CLI에 없어 무용지물.
2. **BLADE 로드맵이 두 제안을 이미 "핵심 요구"로 명문화:**
   - §9 책임 경계: *"의미 검색은 sqlite-vec, 그래프 점프는 codegraph. 둘 다 같은 마크다운에서 동시 인덱싱되어야 영구기관 성립."*
   - §12.1.2: *"backlink 영속화 보강(`edges` 생성 + `doc_links` 역스캔). **영구기관의 핵심 결함.**"* ← 제안 #1의 진짜 필요는 "우선권"이 아니라 **엣지 영속화**
   - embedding_text: *"**concept 노드 노선은 폐기**하고 codegraph_backlinks depth로 재귀 탐색을 표준화."* ← 직전 BLK concept 작업은 BLADE에선 명시적 폐기 대상
3. **제안 #1의 표면적 형태(스티어링)는 CLAUDE.md 도그마상 안 먹힌다.** "Adapt the tool to the agent — don't change the agent": 저살리언스 스티어링(server-instructions·tool desc)은 에이전트의 툴 선택을 못 바꾼다(검증됨). 반면 제안 #2는 *"에이전트가 이미 부르는 툴(query)이, 이미 주는 입력으로 더 많은 일을 하게"* 만드는 것 → 도그마가 "먹힌다"고 분류한 정확한 형태.

## 2. 제안 구조 (To-Be)

### 2.1 구조 다이어그램 (현재 vs 제안)

```
[현재]
agent ──CLI query──> searchNodes ──> nodes_fts(이름만)         → 순수MD에서 []
agent ──MCP only───> codegraph_docs ──> searchDocs(sqlite-vec) → 닿기 어려움
agent ──MCP only───> codegraph_backlinks ──> findBacklinks      → CLI 부재
(BLADE) edges = 0  → callers/callees 무효, §11.2 검증 실패

[제안]
agent ──query "자연어"──> searchNodes(렉시컬)
                            │ thin/zero?
                            └─fallback─> searchDocs(sqlite-vec KNN)  # 제안 #2
                                          + 결과에 backlink 1~2홉 부착  # 제안 #1
index --with-docs ──> doc_links 역스캔 ──> edges 영속화           # 제안 #1 (근본)
(파일명) 한글 ──ASCII 슬러그──> 인덱싱 누락 0                      # 전제 (로드맵 §5.2)
```

### 2.2 새 파일 목록

- (선택) `src/bin/` — CLI `backlinks <file>` / `docs <query>` 서브커맨드 (findBacklinks/searchDocs를 CLI에 노출)
- 신규 파일 불필요가 이상적: 기존 `searchDocs`·`findBacklinks` 재사용. 변경은 **배선(wiring)** 중심.

### 2.3 핵심 API·알고리즘 개념 코드 (의사 코드)

```
# 제안 #2: query 시멘틱 폴백 (codegraph.ts query.action)
results = searchNodes(q, limit)
if docsEnabled(db) and results.length < THIN_THRESHOLD:      # 예: < 3
    sem = await searchDocs(db, q, {topk: limit})             # sqlite-vec KNN
    results = merge(results, sem.hits → node-shape)          # 렉시컬 우선, 시멘틱 보강

# 제안 #1(근본): backlink 엣지 영속화 (indexMarkdown 후처리)
for doc in mdast_metadata where doc_links is not null:
    for link in parseRefs(doc.doc_links):
        target = basenameIndex[link]
        upsert edge(source=docNode(doc), target=docNode(target), kind='doc_link')
# → callers/callees/impact/§11.2 검증이 BLADE에서도 동작
```

## 3. 장단점 비교

### 3-A. 아키텍처 명확성
- **제안 #2**: 세 검색 평면을 "에이전트가 부르는 단일 진입점(query)" 뒤로 통합 → 멘탈모델 단순화. 도그마 정합 ◎.
- **제안 #1(엣지 영속화)**: 그래프 평면을 `mdast_metadata`(읽기 스캔) → `edges`(정식 토폴로지)로 승격 → `callers/callees/impact/trace`가 doc에도 균일 적용. §9 책임 경계 명확화.
- **제안 #1(스티어링 형태)**: 효과 약함(도그마). 단독으론 비추천.

### 3-B. 컴파일·빌드 영향
- 신규 의존성 0. `searchDocs`(sqlite-vec+transformers)는 이미 optionalDependencies + lazy + 게이트. query에서의 호출도 동일 게이트로 best-effort.
- 빌드 자체 영향 없음(배선 변경). dist 재생성 필요.

### 3-C. 현재 작업과의 연관성
- 직전 세션에서 `index -f` ⊇ `--with-docs` + concept 재생성 수정 완료 → 이미 docs 파이프라인을 만지는 흐름의 연속.
- ⚠️ **단, BLADE 로드맵은 concept 노드 폐기를 명시.** 직전 BLK concept 강화는 RX_1(코드+BLK 카탈로그)엔 유효하나 BLADE(순수소설)엔 역방향. → 두 사용처를 가르는 **게이트(코드 유무)** 필요.

### 3-D. 단기 비용 — 위험도

| 위험 | 등급 | 내용 | 완화 |
|---|---|---|---|
| **CLI 콜드스타트 임베딩 지연** | 高 | CLI 일회성 프로세스는 매 `query`마다 MiniLM 모델 로드(~수초). MCP 데몬은 웜 유지하나 CLI는 매번 냉시작 | 시멘틱을 **렉시컬 thin일 때만 폴백**, 또는 `--semantic` 플래그, 또는 순수-MD 인덱스에서만 |
| **core→optional 결합** | 中 | query(always-on core)가 searchDocs(optional deps) 호출 | 기존 게이트(resolveDocsEnabled+loadVec+isEmbedAvailable) 재사용, 미가용 시 무소음 폴백 |
| **concept 폐기와 충돌** | 中 | BLADE는 concept 폐기, RX_1은 concept 사용 | 정책 분기(프로젝트별), §6 RULE과 충돌 시 고위험 명시 |
| **한글 미마이그레이션 시 반쪽** | 中 | ASCII 슬러그 전이 전엔 렉시컬·시멘틱 모두 누락 잔존 | Phase 0에서 마이그레이션 선행 |
| 결과 병합 랭킹 혼란 | 低 | 렉시컬+시멘틱 점수 체계 상이 | 렉시컬 우선, 시멘틱은 보강 표식 |

### 3-E. 장기 유지보수
- 영구기관(§12.4)은 "도구는 교체 가능 부품, 사이클이 본체". query 시멘틱화·엣지 영속화는 **부품 교체가 아니라 부품을 사이클 규격(§9)에 맞추는 것** → 로드맵 수명과 정합.
- 부채 위험: query에 시멘틱 폴백을 넣으면 "query 의미론"이 코드그래프(렉시컬)와 docs(시멘틱) 양쪽을 떠안음 → 문서화·테스트로 의미 고정 필요.

## 4. 순환 의존성·방향 위험 분석

- **방향**: `bin/codegraph(query)` → `docs/search(searchDocs)` → `docs/vec`,`docs/embed`. 단방향, 순환 없음. 단 core CLI가 optional docs 레이어를 호출하는 **하향 결합**이 새로 생김 → 게이트로 격리(미설치 시 no-op)하면 안전.
- **엣지 영속화**: `indexMarkdown`(write-path) 후처리로 `edges` upsert → `graph/`(callers 등)는 변경 없이 수혜. 역방향 의존 없음.
- **위험 방향**: query가 embed를 부르면 CLI 일회성에서 모델 로드가 핫패스로 올라옴 → "조용한 성능 회귀"(CLAUDE.md가 경계하는 부분). 폴백-한정으로 방향 위험 차단.

## 5. 종합 평가 및 추천

### 결론
두 제안은 사변이 아니라 **BLADE 영구기관 로드맵이 이미 명문화한 핵심 요구**다(§9 "의미=sqlite-vec, 그래프=codegraph", §12.1.2 "backlink 엣지 영속화=핵심 결함"). 다만 우선순위는 로드맵 실측 결함에 맞춰야 한다: 제안 #1의 *진짜* 필요는 "우선권 스티어링"(도그마상 안 먹힘)이 아니라 **백링크 엣지 영속화**(BLADE edges=0 해소)이고, 제안 #2는 CLAUDE.md 도그마가 "먹힌다"고 분류한 정확한 형태(*에이전트가 이미 부르는 query를 더 강하게*)라 **최고 레버리지**다. 단, CLI 콜드스타트 임베딩 지연(高)과 한글 미마이그레이션이 전제 조건이라 **Phase 0 선행 없이는 둘 다 반쪽**이 된다.

### 단계별 추천 순서
- **Phase 0 (전제·로드맵 §12.1.1-2)**: ① 한글→ASCII 슬러그 마이그레이션(인덱싱 누락 0) ② `index --with-docs`에서 `doc_links` 역스캔으로 **`edges` 영속화** (BLADE edges=0 → ≥1, §11.2 통과). 이 둘은 코드그래프 변경 없이도 효과 큼.
- **Phase 1 (제안 #2 — 최고 레버리지)**: CLI `query`에 **시멘틱 폴백** 배선 — 렉시컬 결과가 thin/zero이고 docs 활성일 때 `searchDocs` KNN 병합. 콜드스타트는 폴백-한정으로 격리. → task1.md의 빈-쿼리 낭비 직접 해소.
- **Phase 2 (제안 #1 — 표면)**: backlinks/docs를 **CLI 서브커맨드로 노출**(`codegraph backlinks <file>`, `codegraph docs <q>`). 스티어링이 아니라 *툴 표면*으로 우선권 부여. 순수-MD 인덱스 감지 시 query가 그래프 전이 결과를 선두 노출.
- **Phase 3 (정책 분기)**: concept 노드는 BLADE에서 폐기/RX_1에서 유지 — 프로젝트 게이트로 분리.
