# Try — 순수-MD 예외 모드(링크 1등시민화) vs Approach 2 통합 비교

> 작성일: 2026-06-11
> 대상 작업: "순수 마크다운 프로젝트에 한해 옵시디언 링크/백링크를 노드·엣지 1등시민으로 격상"(=Approach A) 와 [[try_001]]의 Approach 2를, codegraph 질의(`node`·`impact`·`callers`·`callees`) 동작 관점에서 비교.
> 코드 구현 없음. 읽기·분석·판단만.

---

## 0. 핵심 결론 먼저 (TL;DR)

- **사용자가 가정한 능력 격차는 존재하지 않는다.** "callers/callees 같은 강력한 코드 질의를 쓰려면 링크를 코드/콜그래프로 위장(masquerade)시켜야 한다"는 전제는 소스 실측 결과 **틀렸다.**
- 실측: **`impact`는 엣지 kind를 가리지 않고**(모든 incoming 엣지 순회, `traversal.js:409`) 동작하고, **`callers`/`callees`/`node`는 `['calls','references','imports']` 화이트리스트**를 순회한다(`traversal.js:191,224`). 즉 **엣지 kind를 `references`로만 줘도** 네 가지 질의가 전부 켜진다.
- ⟹ **두 방식은 "능력"으로 갈리지 않는다.** 갈리는 축은 ① **적용 범위(예외 게이트 vs 상시 가산)**, ② **엣지 kind 선택(코드 위장 vs 정직한 종류)**, ③ **코드 경로 분기 비용** 세 가지뿐이다.
- ⟹ **최적해 = "Approach 2의 메커니즘"을 "Approach A의 게이트(순수-MD 감지)" 아래서 돌리고, 엣지 kind는 정직하게 `references`로 두되 필요 시 `calls` 위장을 옵션화**하는 하이브리드. A와 2는 라이벌이 아니라 **"언제 켜지나(A)" × "어떻게 만드나(2)"** 의 직교 축이다.

## 1. 현황 (As-Is)

### 1.1 파일 구조

[[try_001]]에서 확립된 사실을 승계한다(중복 최소화):
- 대상 프로젝트 `D:\Unity\BLADE` = 순수 문서(42 md, 코드 0). `mdast_metadata` 42행, `doc_links` 107개. `nodes`/`edges` 0행.
- 도구 `@evespimrose/codegraph@0.9.8.1`, 전역 설치는 `dist/`만(소스 없음).
- 본 분석이 새로 파고든 곳: **질의 측 코드** — `dist/graph/traversal.js`, `dist/graph/queries.js`, `dist/db/queries.js`, `dist/mcp/tools.js`.

### 1.2 병목·문제 정량화 (질의별 엣지 kind 필터 — 실측)

각 질의가 **어떤 edge kind를 순회하는지**를 소스에서 직접 확인했다. 이것이 두 방식 비교의 토대다.

| 질의(MCP 도구) | 순회 대상 엣지 | 필터 근거(file:line) |
|---|---|---|
| `impact` (`getImpactRadius`) | **모든 incoming 엣지(kind 무관)** + outgoing `governs` | `traversal.js:409`(필터 없음), `:421`(governs) |
| `callers` (`getCallers`) | incoming **`calls`/`references`/`imports`** | `traversal.js:191` |
| `callees` (`getCallees`) | outgoing **`calls`/`references`/`imports`** | `traversal.js:224` |
| `node` 트레일 (`formatTrail`) | callers+callees와 동일 | `tools.js:3168-3169` |
| `trace`/`explore` 이웃 확장 | **`calls`만**(엄격) | `tools.js:2210` (`e.kind!=='calls'` skip) |
| 메트릭 `callCount` | **`calls`만**(엄격) | `queries.js:269-270` |
| `findDeadCode` | incoming 중 `contains` 제외(나머지 전부) | `queries.js:300` |
| `backlinks` (`findBacklinks`) | **엣지 안 봄** — `mdast_metadata.doc_links` JSON 직접 | `search.js:190` |

엣지 kind 파라미터 의미도 확인: `getIncomingEdges/​getOutgoingEdges(id, kinds)`는 `kinds`가 있으면 `WHERE kind IN (...)`, 없으면 **전부 반환**(`db/queries.js:1062-1097`).

### 1.3 결정적 발견 (질의 커버리지 매트릭스)

위 표를 "엣지 kind를 무엇으로 격상하느냐" 관점으로 뒤집으면, **선택한 kind별로 어떤 질의가 켜지는지**가 한눈에 나온다. 이게 이 문서의 핵심 산출물이다.

| 질의 | `calls`(위장) | `references`(정직) | `imports` | 신규 `links` |
|---|:---:|:---:|:---:|:---:|
| **`impact`** | ✅ | ✅ | ✅ | ✅ (kind 무관) |
| **`callers`** | ✅ | ✅ | ✅ | ❌ (화이트리스트 수정 필요) |
| **`callees`** | ✅ | ✅ | ✅ | ❌ |
| **`node` 트레일** | ✅ | ✅ | ✅ | ❌ |
| `trace`/`explore` | ✅ | ❌ | ❌ | ❌ |
| 메트릭 callCount | ✅ | ❌ | ❌ | ❌ |
| `backlinks` | 무관(JSON) | 무관 | 무관 | 무관 |

**판독:**
1. 사용자가 명시한 **4대 질의(node·impact·callers·callees)** 는 **`references` 한 종류로 전부 충족**된다. → "코드로 위장해야 한다"는 전제 폐기.
2. `calls`(위장)를 쓰면 추가로 `trace`/`explore`/`metrics`까지 켜진다. **위장의 유일한 실익은 이 세 질의.** 대가는 의미 오염(문서가 "함수 호출"로 분류됨, dead-code 분석에 문서가 섞임 등).
3. 완전 신규 `links` kind는 `impact`만 켜지고 callers/callees는 죽는다(화이트리스트 `['calls','references','imports']`에 없음). → 신규 kind를 쓰려면 **도구 코드 1줄 수정**(화이트리스트에 추가)이 동반돼야 한다.
4. `references`는 **이미 코드용으로 쓰이는 kind**다(타입 참조, 이벤트 핸들러 — `tree-sitter.d.ts:221`, `dfm-extractor.js:136`). 순수-MD 프로젝트에선 충돌이 없지만, **혼합 프로젝트에선 문서 링크와 코드 타입참조가 같은 kind로 섞인다**(§3-D 중위험).

## 2. 제안 구조 (To-Be)

### 2.1 구조 다이어그램 (Approach A vs Approach 2)

두 방식 모두 `doc_links`(이미 추출됨) → `nodes`+`edges` 승격이라는 **동일 메커니즘**을 공유한다. 차이는 **게이트 위치**와 **kind 선택**.

```
공통 메커니즘 (try_001 §2):
  doc_links(JSON) ──승격패스──> nodes(doc) + edges(?kind?)
                                        │
                       ┌────────────────┴────────────────┐
                  node/impact/callers/callees 가 여기서 읽음

[Approach A — 순수-MD 예외 게이트]
  indexAll 시작
     └─ 프로젝트 타입 감지: 코드노드==0 && mdast_metadata>0 ?
            ├─ YES(순수-MD) → 승격 패스 ON, kind='calls'(위장) → 전체 질의 표면 점령
            └─ NO(코드 포함) → 승격 패스 OFF (코드 그래프 무손상)

[Approach 2 — 상시 가산(플래그)]
  indexAll 시작
     └─ CODEGRAPH_DOCS(or 신규 플래그) ON ?
            ├─ YES → 승격 패스 ON, kind='references'(정직), 코드/문서 공존
            └─ NO  → 패스 OFF
   ※ 코드 포함 프로젝트에서도 켜지면 문서·코드 엣지가 한 그래프에 공존
```

### 2.2 새 파일 / 수정 지점 목록

| 항목 | Approach A (예외 모드) | Approach 2 (가산) |
|---|---|---|
| 승격 패스(신규) | `doc-graph-linker.ts` (~70~100줄) | `doc-links-linker.ts` (~60~90줄, try_001) |
| 문서 노드 upsert | indexer 내 +5~10줄 | 동일 |
| 호출 지점 | indexAll + **프로젝트 타입 감지 분기** | indexAll + 플래그 체크 |
| **프로젝트 타입 감지기(신규)** | **필요** — `isPureMarkdownProject()` (~15~25줄) | 불필요 |
| 엣지 kind | `calls`(위장) 또는 설정값 | `references` |
| 화이트리스트 수정 | 불필요(`calls`는 이미 포함) | 불필요(`references` 포함) |
| status 표기(선택) | "Markdown-as-code mode" 라벨 | "Doc nodes/links" 카운트 |

> 두 방식의 신규 코드량은 비슷(±20줄). **A가 "타입 감지기"만큼 더 든다.**

### 2.3 핵심 알고리즘 개념 (의사 코드 — 구현 아님)

```text
// 공통: 승격 패스 (try_001 §2.3와 동일, basenameIndex/parseRefs는 search.js에서 재사용)
function promoteDocLinks(db, edgeKind):
    upsert each doc as node(kind='doc', name=basename, qualified_name=rel)
    for each (srcDoc, link) in mdast_metadata.doc_links:
        target = basenameIndex[link] ?? null
        if target and target != srcDoc:
            insertEdge(src→target, kind=edgeKind, provenance='mdast-link')

// Approach A 전용: 게이트 + 위장
function isPureMarkdownProject(db):
    codeNodes = SELECT count(*) FROM nodes WHERE kind NOT IN ('doc','concept')
    docs      = SELECT count(*) FROM mdast_metadata
    return codeNodes == 0 && docs > 0

onIndexAll():
    if isPureMarkdownProject(db):
        promoteDocLinks(db, edgeKind = config.docEdgeKind ?? 'calls')  // 위장→전체표면
    // else: 손대지 않음

// Approach 2 전용: 플래그 + 정직
onIndexAll():
    if resolveDocsEnabled(db):                      // 기존 CODEGRAPH_DOCS 재사용 가능
        promoteDocLinks(db, edgeKind = 'references') // node/impact/callers/callees 충족
```

핵심: `promoteDocLinks`는 동일. **A는 `isPureMarkdownProject()` 게이트 + `calls` 위장**, **2는 플래그 게이트 + `references` 정직**. 나머지는 같다.

## 3. 장단점 비교

### 3-A. 아키텍처 명확성

| 기준 | Approach A (예외 모드) | Approach 2 (가산) |
|---|---|---|
| 의미 정합성 | 문서를 "코드"로 위장(`calls`) → 출력에 "calls/함수" 어휘가 문서에 붙음. 부정직하나 **격리됨**(순수-MD에서만) | `references`로 정직. 단 코드 kind와 **공유**되어 혼합 프로젝트에선 구분 모호 |
| 모드 명료성 | "이 프로젝트는 MD-그래프 모드" 라고 단정 가능 → 멘탈모델 단순 | 항상 "코드+문서 혼합 그래프" → 결과 해석 시 둘을 분별해야 |
| 질의 표면 | **전체 점령**(trace/explore/metrics까지, 위장 덕분) | 4대 질의(node/impact/callers/callees)만. trace/explore는 안 잡힘(`references`라서) |
| 진입점 문제 해소 | doc 노드 42개가 곧 진입점. status도 비지 않음 | 동일 |

⟹ **순수-MD 프로젝트 한정**이라면 A가 "모드가 명확하고 질의 표면이 넓다"는 점에서 깔끔. 단 의미 부정직(문서=calls)이 영구 부채.

### 3-B. 컴파일·빌드 영향

- 공통: 별도 소스 저장소에서 TS 수정 → 빌드 → 재설치/버전범프(전역 dist 직접 패치 비권장). [[try_001]] §3-B와 동일.
- A 추가 비용: **프로젝트 타입 감지기**가 indexAll 흐름에 분기를 추가. 분기는 단순하나(코드노드 카운트), "감지 오판" 시 모드가 통째로 바뀌는 만큼 테스트 부담↑.
- 스키마: 둘 다 `kind` 자유 텍스트라 마이그레이션 불필요.
- 화이트리스트: A의 `calls`·2의 `references` 모두 기존 `['calls','references','imports']`에 포함 → **둘 다 도구 화이트리스트 수정 불요**. (단 신규 `links` kind 채택 시에만 `traversal.js:191,224` 1줄 수정 필요.)

### 3-C. 현재 작업과의 연관성

- WORKFLOW §9 표가 약속한 "codegraph = 정/역 + 재귀 백링크 + impact"의 공백을 **둘 다 메운다.** 추가로 사용자가 명시한 callers/callees도 **둘 다 충족**(`references`/`calls` 모두 화이트리스트 내).
- WORKFLOW §2.2 Step 2(그래프 전이)는 지금 `backlinks`(JSON 경로)로 이미 동작 → 두 방식은 **그 위에 node/impact/callers/callees를 더하는 것**이지 백링크를 대체하지 않는다.
- [[try_001]]의 권고(Approach 2)는 본 문서에서 **"게이트 방식과 kind 선택"이라는 두 자유도로 분해**된다. 즉 try_001의 Approach 2는 "플래그 게이트 + references"라는 한 점이었고, Approach A는 "타입감지 게이트 + calls"라는 다른 점이다.

### 3-D. 단기 비용 — 위험도 분류

| 위험도 | 항목 | 해당 방식 | 설명/완화 |
|---|---|---|---|
| 🔴 고위험 | 전역 패키지 포크 유지(재설치 시 dist 소실) | 공통 | 소스 저장소 수정→0.9.9 범프(소유자라 가능) |
| 🔴 고위험 | **`calls` 위장의 의미 오염 영구화** | **A** | dead-code/metrics/trace가 문서를 "함수 호출"로 취급. 순수-MD에 격리되나, 나중에 코드가 1개라도 추가되면 모드가 꺼지며 그래프 의미가 급변(아래 방향위험) |
| 🟡 중위험 | **혼합 프로젝트에서 문서 `references`가 코드 타입참조와 섞임** | **2** | callers/callees 결과에 문서 링크가 코드 참조와 혼재. → 신규 `links` kind+화이트리스트 수정으로 분리(단 callers/callees 재활성 위해 1줄 추가 필요) |
| 🟡 중위험 | **프로젝트 타입 감지 오판** | **A** | 코드 0인데 감지 실패→모드 안 켜짐 / 코드 추가됐는데 캐시로 순수-MD 유지. → 감지를 매 indexAll 재평가 + 명시적 override 플래그 |
| 🟡 중위험 | 모드 전환 시 그래프 재구성 비용 | A | 순수→혼합 전환 시 `calls`(문서) 엣지 전량 폐기·재계산 필요 |
| 🟢 저위험 | 사이클/중복 링크 | 공통 | visited-set + (src,tgt,kind) 유니크 (try_001 §4) |
| 🟢 저위험 | 한글/한자 파일명 | 공통 | basename 매칭 기해결(실측) |
| 🟢 저위험 | `node` 미해석 | 공통 | `findSymbol`은 kind 비제한(`tools.js:3115`) → doc 노드 해석됨 |

### 3-E. 장기 유지보수

- **A**: "순수-MD 모드"라는 **별도 코드 경로**가 영구히 분기로 남는다. 코드↔문서 경계가 흐려지는 프로젝트(예: BLADE에 도구 스크립트가 생기는 순간)에서 모드가 토글되며 그래프 의미가 요동친다. 위장(`calls`) 어휘가 출력에 박혀 사용자 혼란 소지.
- **2**: 단일 경로. 문서 그래프가 항상 같은 규칙으로 코드 그래프 옆에 공존. 혼합 프로젝트로 자연 확장. 단 "문서 references vs 코드 references" 분별을 위해 결국 전용 `links` kind로 가는 게 깔끔(그러면 화이트리스트 1줄 수정).
- 범용성: **2가 더 미래지향**(코드+문서 양용). A는 "코드 없는 볼트" 전용 최적화.

## 4. 순환 의존성·방향 위험 분석

- **빌드 의존성**: 두 방식의 승격 패스 모두 `indexer`(노드 헬퍼)+`search`(basenameIndex)에만 의존 → `governs-linker → indexer` 선례와 동형, 상향 순환 없음([[try_001]] §4).
- **A의 모드 토글 = 상태 의존 위험**: `isPureMarkdownProject()` 결과가 **DB 상태(코드노드 수)에 의존**하므로, 인덱싱 순서/캐시에 따라 같은 프로젝트가 다른 모드로 평가될 여지. → 감지를 "이번 indexAll의 최종 코드노드 수"로 **인덱싱 종료 후 1회 평가**(중간 상태로 판단 금지)해야 안전. 이건 A 고유의 방향성 함정.
- **엣지 방향**: link(인용) → target(피인용)으로 단방향. `impact`(역방향 incoming)·`callers`(incoming)·`callees`(outgoing) 모두 방향 일관. governs가 "concept→code"인 것과 대칭이라 기존 traversal과 호환(실측).
- **이중 경로(공통, 저위험)**: 승격 후에도 `backlinks`는 여전히 JSON을 읽음 → 백링크 정보가 JSON과 edges 두 곳에 상주. 둘 다 같은 `doc_links` 파생이라 모순 위험 낮음. 장기 통일은 선택(try_001 Phase 4).

## 5. 종합 평가 및 추천

### 결론

**A와 Approach 2는 경쟁 설계가 아니라 직교하는 두 자유도다 — A는 "언제 켜지나(순수-MD 게이트)", 2는 "어떻게 만드나(doc_links→edges 승격)".** 그리고 사용자가 둘을 가른 기준("강력한 코드 질의를 쓰려면 위장이 필요하다")은 **실측상 성립하지 않는다**: `impact`는 kind를 안 가리고, `callers`/`callees`/`node`는 `references`까지 순회하므로(`traversal.js:191,224,409`) **정직한 `references` 엣지만으로 4대 질의가 전부 켜진다.** `calls` 위장의 유일한 추가 실익은 `trace`/`explore`/`metrics`인데, 그 대가로 "문서가 함수 호출로 분류되는" 영구 의미 부채를 진다. 따라서 권고는 **하이브리드**다: **Approach 2의 승격 메커니즘**을 채택하되, **Approach A의 순수-MD 게이트로 감싸** 코드 프로젝트 회귀를 원천 차단하고, 엣지 kind는 **정직하게 `references`**(또는 전용 `links`+화이트리스트 1줄 추가)로 두고, `calls` 위장은 "trace/explore까지 원하면 켜는" **명시적 옵트인 설정**으로만 남긴다.

### 단계별 추천 순서

- **Phase 0 — 결정 (코드 0)**
  - 게이트: 순수-MD 자동감지(A) vs 플래그(2) vs **둘 다(자동감지 + override 플래그)** ← 권고.
  - 엣지 kind: **`references`(권고, 4대 질의 충족)** / `links`(분리하려면 화이트리스트 1줄) / `calls`(전체표면 원할 때만 옵트인).
  - 노드 kind: `doc`(권고) — `findSymbol`이 해석 가능(`tools.js:3115`).

- **Phase 1 — 승격 메커니즘 (A·2 공통)**
  - 문서 노드 upsert + `doc-links-linker` 후처리(try_001 §2.3). basenameIndex/parseRefs는 `search.js`에서 추출·공유.
  - 검증: `nodes`≈42, `edges`≈107, `codegraph_node "인물_강은휘"` 트레일에 백링크 문서가 뜨는지.

- **Phase 2 — 게이트 적용 (A의 가치 흡수)**
  - `isPureMarkdownProject()` = (코드노드 0 && mdast>0), **indexAll 종료 후 1회 평가**(§4 함정 회피).
  - override 환경변수(예: `CODEGRAPH_DOC_GRAPH=1/0`)로 자동감지 무시 허용.
  - 검증: 코드 더미 1개 추가 시 모드 OFF로 떨어지고 코드 그래프 출력이 byte-identical인지.

- **Phase 3 — 4대 질의 회귀**
  - `node`/`impact`/`callers`/`callees`를 BLADE 문서 5개에 대해 실행 → 1~2홉 폐쇄 확인(WORKFLOW §8 T2).
  - `references` 채택 시 `trace`/`explore`는 비는 게 정상임을 문서화(기대치 합치).

- **Phase 4 — 선택 정리**
  - `calls` 위장 옵트인 설정 추가(원하면 trace/explore/metrics까지).
  - `findBacklinks`를 edges 기반으로 통일(JSON/edges 이중 경로 제거).
  - WORKFLOW §5·§12의 슬러그/마이그레이션 권고 철회 반영([[try_001]] §5).
